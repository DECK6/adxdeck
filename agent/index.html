01 · Orientation

수업 개요와 성공 기준

오전에는 Ableton MCP의 마지막 실습을, 오후에는 별도의 Hermes Agent·AKM 입문을 진행합니다. 두 환경의 기술은 독립적이지만, 요청·실행·확인이라는 학습 방식은 공유합니다.

Ableton MCP Ableton Live의 현재 상태를 조회하고, 구현이 지원하는 범위에서 트랙·클립·매개변수를 다루는 연결입니다.
≠

04 · Module A

Ableton MCP 최종 실습

연결 표시만 보는 데서 끝내지 않습니다. 실제 상태를 조회하고 화면과 대조한 다음, 작은 결과를 만들고 세 번 고쳐 저장합니다.

연결 확인→ 현재 상태 조회→ 생성→ 귀로 확인→ 수정 3회→ 저장·제출

도구 역할을 먼저 구분하기

구성요소이번 실습의 역할보장하지 않는 것
Ableton Live트랙·클립·악기·오디오·tempo·재생·저장센서 원본 수집의 기본 도구는 아닙니다.
Ableton MCP에이전트가 Live 상태를 조회하고 가능한 작업을 요청하는 연결음악적 취향과 완성도를 자동으로 보장하지 않습니다.
TouchDesigner센서 값 시각화, 신호 전처리, 다른 시스템으로 라우팅음악 제작 전체를 대신하지 않습니다.
OSC센서·제어 값을 주소와 값 형태로 전달그 자체가 음악을 생성하지 않습니다.
MIDI노트, velocity, CC 등 음악 장치 중심 데이터 전달모든 센서의 의미를 자동으로 이해하지 않습니다.

1. 연결 순서

  1. Ableton Live를 완전히 실행합니다.
  2. 계정·라이선스 경고가 없는지 확인합니다.
  3. Preferences의 Control Surface 상태를 확인합니다.
  4. 사용 중인 Ableton MCP server, bridge 또는 connector를 실행합니다.
  5. MCP 클라이언트를 실행하거나 연결을 새로 고칩니다.
  6. 도구 목록에 Ableton 관련 도구가 보이는지 확인합니다.
  7. 아무것도 바꾸지 않는 조회 요청으로 연결을 시험합니다.
구현별 차이Ableton MCP의 버전과 구현에 따라 메뉴명·server 실행 방식·도구 이름·지원 기능이 다릅니다. 화면에 없는 명령을 추측하지 말고 현재 도구 목록을 기준으로 진행하세요.

2. 현재 Live Set을 변경 없이 조회하기

프롬프트 A1 · 현재 상태 감사
현재 열린 Ableton Live Set을 변경하지 말고 조회만 해줘.
1) Set 이름
2) tempo/BPM
3) 트랙 이름과 종류
4) 현재 있는 clips 또는 scenes
5) 현재 재생 상태
를 표로 정리해줘. 조회할 수 없는 항목은 추측하지 말고 "확인 불가"라고 써줘.

3. 안전한 첫 수정

프롬프트 A2 · tempo 하나만 수정
현재 Set의 구조를 먼저 다시 확인해줘.
그 다음 tempo만 112 BPM으로 바꾸고, 다른 트랙·클립·장치·재생 상태는 건드리지 마.
변경 전 값과 변경 후 값을 보고해줘.
PASS 증거실제 tempo가 112로 바뀌고, 다른 항목에는 의도하지 않은 변화가 없으며, 보고 내용과 Live 화면이 일치합니다.

4. 8–16마디 루프 생성

프롬프트 A3 · 미니멀 전자음악 루프
현재 Set을 먼저 조회하고 기존 작업을 덮어쓰지 않는 방식으로 진행해줘.
112 BPM, 4/4 박자, 8마디 길이의 미니멀 전자음악 루프를 만들어줘.
가능하면 다음 역할을 분리해줘.
- Kick
- Bass
- Pad 또는 Texture
- FX 또는 Percussion
각 트랙과 clip의 이름에 역할이 드러나게 하고, 만든 뒤 실제 생성한 항목을 목록으로 보고해줘.
지원하지 않는 작업은 임의로 성공했다고 말하지 말고 수동 대안을 알려줘.

16마디를 선택하면 위 프롬프트의 8마디를 16마디로 바꿉니다. 새 트랙 또는 clip이 실제로 생겼는지, 길이가 맞는지, 소리가 나는지 확인하세요. 악기가 할당되지 않았다면 그 상태도 정확히 기록합니다.

5. 자연어 수정 3회

한 번에 한두 변수만 바꿔야 전후 차이를 들을 수 있습니다.

프롬프트 A4 · 밀도
방금 만든 루프에서 Kick은 유지하고, Bass note의 밀도만 절반 정도로 줄여줘.
다른 트랙은 바꾸지 말고, 무엇을 수정했는지 전후 차이를 설명해줘.
프롬프트 A5 · 공간과 대비
Pad 또는 Texture 트랙을 더 어둡고 넓게 들리게 조정해줘.
가능한 범위에서 한 가지 장치 또는 parameter만 바꾸고, 변경한 parameter 이름과 값을 알려줘.
지원하지 않으면 수동 조정 위치만 안내해줘.
프롬프트 A6 · 후반 구조
8마디 후반 2마디에만 작은 변화를 만들어줘.
새 요소를 무작정 추가하지 말고, 기존 Percussion 또는 FX의 반복·mute·velocity 중 가능한 한 가지 방식으로 대비를 만들어줘.
변경 후 전체 구조를 1–8마디 기준으로 설명해줘.

05 · Clear separation

Ableton MCP ≠ Hermes

Ableton MCP

특정 창작 앱을 AI가 다룰 수 있게 연결합니다. Ableton Live와 해당 MCP 구현이 필요합니다.

Hermes Agent

여러 도구와 로컬 파일을 사용해 목표를 수행하는 독립 환경입니다. 별도로 설치하며 Ableton MCP가 없어도 작동합니다.

AKM

작업 자료·정리된 지식·다음 행동·공유 결과를 폴더와 문서에 남겨 다시 읽을 수 있게 하는 방식입니다.

공통 학습 포인트두 환경 모두 자연어 요청 → 도구 실행 → 실제 결과 확인 → 재요청의 고리를 사용합니다. 차이는 대상 시스템과 필요한 구성입니다.

06 · Module B

Hermes Agent 기초

이번 범위는 한 provider·한 model·한 작업 폴더에서 정상 대화와 파일 읽기·쓰기를 확인하는 것입니다. 먼저 기본 대화를 안정화한 뒤 도구를 점검합니다.

챗봇·코딩 에이전트·Hermes 비교

구분중심 인터페이스주요 산출물이번 실습의 관점
챗봇대화답변설명과 아이디어를 빠르게 얻습니다.
코딩 에이전트프로젝트·코드 폴더코드·파일 변경이미 경험한 도구와 파일 작업 방식을 비교합니다.
Hermes AgentCLI/TUI와 연결 가능한 실행 표면파일·명령·검색·도구 호출을 포함한 결과provider 선택, tools, skills, memory 등으로 확장 가능한 프레임워크 중 로컬 파일 흐름만 익힙니다.

model만 선택·변경할 때

provider/model 설정
hermes model
  • provider는 모델을 제공하고 인증·요금·사용 한도를 관리하는 경로입니다.
  • model은 실제 대화와 도구 계획을 수행합니다.
  • 인증 방식과 비용은 provider마다 다릅니다. 로그인 정보는 본인 화면에서만 입력하세요.
  • model 이름과 provider 인증이 맞지 않으면 hermes model을 다시 실행합니다.
  • local/custom model은 최소 context 요구사항을 공식 문서에서 별도로 확인합니다.

07 · Module C

Hermes–AKM 최소 흐름

AKM은 ‘AI가 알아서 기억한다’고 기대하는 대신, 원본·정리된 지식·다음 행동·외부 산출물을 구분된 폴더와 파일로 남겨 다음 세션이 다시 읽을 수 있게 하는 작업 방식입니다.

1. starter 구조

hermes-akm-starter/
├── AGENTS.md
├── README.md
├── 00-inbox/
├── 20-knowledge/
├── 60-actions/
└── 80-outputs/

00-inbox

아직 분류·정리가 끝나지 않은 원본과 캡처

20-knowledge

원본에서 뽑아 출처와 함께 정리한 재사용 지식

60-actions

다음 행동·결정·완료 증거

80-outputs

다른 사람에게 보여주거나 제출할 결과

2. 프로젝트 규칙 파일 예시

AGENTS.md는 이 프로젝트 안에서 에이전트가 따라야 할 작업 규칙을 남기는 파일입니다. 이번 실습에서는 매 작업의 시작에 명시적으로 읽게 합니다.

AGENTS.md 예시
# Project Rules

- Work only inside this folder.
- Read existing files before creating a new file.
- Put raw captures in `00-inbox/`.
- Put reusable synthesized notes in `20-knowledge/`.
- Put next actions and decisions in `60-actions/`.
- Put final shareable results in `80-outputs/`.
- Never write passwords, API keys, tokens, or private personal data.
- After writing, read the file back and report its exact path.
- Do not claim success if the file does not exist.

3. 구조를 읽고 계획만 세우기

프롬프트 C1 · 폴더 역할과 4단계 계획
현재 폴더의 AGENTS.md와 하위 폴더 목록을 읽어줘.
각 폴더의 역할을 한 문장으로 설명하고, 파일을 만들기 전에 오늘 작업 계획을 4단계로 제시해줘.
계획만 보여주고 아직 수정하지 마.

실제 폴더 이름을 읽었는지, 없는 폴더를 만들었다고 가정하지 않았는지, 계획이 inbox → knowledge → action → output 순서와 맞는지 확인합니다.

4. 원본과 해석 분리하기

프롬프트 C2 · inbox에서 knowledge로
00-inbox/sample-source.md와 00-inbox/first-capture.md를 읽어줘.
원문에 없는 사실을 추가하지 말고, 두 파일에서 반복되는 핵심 원칙만 정리해서
20-knowledge/hermes-first-principles.md를 만들어줘.
각 원칙 아래에는 어느 source file에서 가져왔는지 파일명을 써줘.
완료 후 source와 해석이 섞이지 않았는지 점검해줘.
PASS 증거원본 파일은 그대로 남고, knowledge 문서에는 근거 파일명이 표시되며, 출처에 없는 기능을 사실처럼 추가하지 않습니다.

5. 다음 행동 만들기

프롬프트 C3 · 30분 행동 3개
20-knowledge/hermes-first-principles.md를 읽고,
다음 수업 전 30분 안에 할 수 있는 행동 3개를 정해
60-actions/next-actions.md에 체크리스트로 만들어줘.
각 행동에는 완료 증거를 한 줄씩 붙여줘.

완료 증거는 파일 경로, 스크린샷, hermes doctor의 오류 여부, readback 확인 문장처럼 관찰할 수 있어야 합니다.

6. 공유 가능한 output 만들기

프롬프트 C4 · 1페이지 요약
지금까지 만든 00-inbox, 20-knowledge, 60-actions의 관련 파일을 읽어줘.
다른 학생에게 보여줄 수 있는 1페이지 요약을
80-outputs/my-hermes-akm-summary.md에 만들어줘.
다음 구조를 지켜줘.
1. 오늘 한 일
2. 확인된 결과
3. 실패하거나 수동으로 처리한 부분
4. 다음 행동
5. 사용한 source file 목록
완료 후 파일을 다시 읽고, 개인정보나 credential 값이 없는지 점검해줘.

7. 새 세션에서 복원하기

  1. 현재 작업을 저장합니다.
  2. Hermes를 정상 종료합니다.
  3. 같은 starter 폴더에서 새 Terminal 또는 PowerShell을 엽니다.
  4. hermes를 다시 실행합니다. 현재 버전의 도움말에 /new가 보일 때만 해당 명령을 사용할 수 있습니다.
프롬프트 C5 · 근거 기반 새 세션 복원
이 세션은 이전 대화를 모른다고 가정해줘.
현재 작업 폴더의 AGENTS.md를 먼저 읽고,
00-inbox, 20-knowledge, 60-actions, 80-outputs에서 오늘 만든 Markdown 파일을 찾아 읽어줘.
그 근거만 사용해 다음을 보고해줘.
- 지금까지 완료된 작업
- 확인된 결과
- 실패 또는 미검증 상태
- 다음 행동 3개
- 근거로 읽은 파일 경로
추측한 내용은 따로 표시해줘.
프롬프트 C6 · 복원 결과의 근거 점검
방금 복원한 내용 중 source file에 직접 근거가 없는 문장을 찾아줘.
근거가 없으면 삭제하거나 "추정"으로 표시해줘.
그 다음 60-actions/next-actions.md의 첫 번째 미완료 항목만 실행 계획으로 바꿔줘.
아직 실제 실행은 하지 마.
복원 PASS새 세션이 기존 파일을 실제로 읽고, 완료·실패·다음 행동을 구분하며, 근거 파일 경로를 제시하고, 과거 대화를 자동으로 기억한다고 과장하지 않습니다.

12 · Reflection

Exit Ticket

3분 동안 다음 세 문항에 답합니다.

  1. 오늘 AI가 실제로 바꾼 것 하나와 내가 직접 확인한 증거는 무엇인가?
  2. Ableton MCP와 Hermes Agent가 기술적으로 다른 이유를 한 문장으로 설명하라.
  3. 새 세션이 이전 작업을 복원하려면 어떤 파일과 규칙을 먼저 읽어야 하는가?

선택 문항

  • 음악적 판단을 AI에게 맡기지 않아야 했던 순간은 언제였는가?
  • 파일을 잘못된 폴더에 만들었을 때 어떤 기준으로 고쳤는가?
  • 다음 수업 전 30분 안에 완료할 행동은 무엇인가?

13 · Compact routes

90분·180분 축약 안내

설치가 이미 완료된 상황에서 사용할 수 있는 축약 흐름입니다. 설명보다 실행 고리를 우선 보존합니다.

90분 흐름
시간내용반드시 남길 것
0–10분두 환경 구분·연결 진단구분 문장
10–35분Ableton 조회 → 8마디 루프 → 수정 2회.als·수정 로그
35–45분저장·비평음악적 판단 문장
45–60분Hermes 첫 실행·작업 폴더 확인정상 응답
60–75분first-capture.md 생성·readback실제 파일
75–88분새 세션에서 규칙·파일 복원근거 기반 보고
88–90분Exit Ticket세 문항

수정은 2회로 줄이고, AKM은 00-inbox와 80-outputs를 직접 사용하되 전체 4폴더 역할은 구두로 설명합니다.

공식 문서를 우선하세요설치 명령·메뉴·provider 인증·모델 목록은 바뀔 수 있습니다. 페이지의 안내와 현재 화면이 다르면 공식 문서와 설치된 버전의 도움말을 먼저 확인하세요.

반드시 알고 시작할 한계

  • Ableton MCP는 구현과 버전에 따라 도구 이름과 지원 기능이 다릅니다. 자연어 프롬프트는 목표를 정의하지만 실제 가능 범위는 현재 도구 목록으로 확인해야 합니다.
  • Ableton Live 계정·라이선스·기관 계정 정책은 각자의 공식 계정 화면에서 확인해야 합니다.
  • Windows native, WSL2, macOS는 shell·홈 폴더·권한 방식이 다릅니다. 명령과 경로를 서로 섞지 마세요.
  • provider의 모델 이름·무료 범위·OAuth 동작·context limit는 변경될 수 있습니다. hermes model 화면과 provider 안내를 확인하세요.
  • Hermes의 tools·skills·memory 등은 설치와 설정에 따라 달라집니다. 이번 완료 기준은 한 provider/model의 정상 응답과 지정 폴더 안의 파일 작업입니다.
  • qmd 같은 의미 기반 검색 도구는 문서가 많아졌을 때 선택할 수 있습니다. 작은 starter에서는 파일명을 확인하고 직접 읽는 방식으로 충분합니다.

한 장으로 복습하기

Ableton MCP

  1. Live를 먼저 엽니다.
  2. 라이선스와 Control Surface를 봅니다.
  3. MCP connector와 도구 목록을 봅니다.
  4. 현재 Set을 조회합니다.
  5. 작은 루프를 만듭니다.
  6. 한두 변수씩 고칩니다.
  7. 귀와 화면으로 확인합니다.
  8. 새 이름으로 저장합니다.

Hermes Agent

  1. starter 폴더에서 shell을 엽니다.
  2. hermes --version과 hermes doctor를 실행합니다.
  3. 처음에는 hermes setup을 사용합니다.
  4. hermes 또는 hermes --tui로 실행합니다.
  5. 파일보다 먼저 현재 폴더와 규칙을 읽게 합니다.
  6. 생성 후 정확한 경로와 내용을 다시 읽게 합니다.
  7. 응답보다 실제 파일을 믿습니다.
