§ 01 핵심 결론

TL;DR — Three Lines
  • AI Overviews / AI Mode 는 기존 Search 위에 RAG(그라운딩) + query fan-out 을 얹은 구조 — 그래서 기존 SEO 기본기가 그대로 핵심 방법이다.
  • “AEO / GEO” 는 Google 입장에서 별도 분야가 아니라 SEO 의 연장선. llms.txt, 강제 chunking, AI 전용 재작성, 가짜 mention 만들기는 무효이거나 정책 위반.
  • 실행 방법은 4축: ① 비범용 콘텐츠, ② 기술 구조, ③ 로컬·이커머스 신호, ④ agentic 경험 대응.

§ 02 AI 검색이 동작하는 방식

R
Retrieval-Augmented Generation (RAG / grounding)
기존 Search 인덱스에서 관련 페이지를 검색 → 그 페이지의 정보를 근거로 응답 생성 → 응답 안에 인용 가능한 클릭형 링크로 노출.
Q
Query fan-out
모델이 사용자 쿼리 1개에서 관련 보조 쿼리 N개를 동시 생성해 추가 결과를 가져옴.
예: “잡초가 가득한 잔디 고치는 법” → “잔디용 제초제 추천”, “약품 없이 잡초 제거”, “잡초 발생 예방” 등.

함의. AI 응답에 등장하려면 인덱싱 + 스니펫 표시 자격이 우선이고, fan-out 으로 생성되는 다양한 의도 변형에도 의미적으로 매칭될 수 있는 콘텐츠여야 한다.

§ 03 방법 — 콘텐츠 (Non-commodity, People-first)

  1. 고유한 관점 제공. 직접 경험 기반 1차 리뷰·관찰을 담는다. 재요약, 생성형 AI 로 쉽게 만들 수 있는 글은 피한다.
  2. Commodity → Non-commodity 전환.
    • ✗ “처음 집을 사는 사람을 위한 팁 7가지” — 공통지식 기반 commodity.
    • ✓ “왜 우리는 인스펙션을 포기하고 돈을 아꼈는가 — 하수관 라인 내부 사례” — 전문성·체험 기반 non-commodity.
  3. 사람이 읽기 좋은 구조화. 단락·섹션 분할, 명확한 헤딩, 자연스러운 탐색 흐름.
  4. 이미지·비디오 보강. 텍스트를 보완하는 고품질 미디어를 적절한 지점에 배치. 기존 Image SEO / Video SEO 가이드 준수가 곧 AI 검색 최적화.
  5. 변형 폭증 금지. fan-out 변형마다 별도 페이지를 양산하면 scaled content abuse 정책 위반 위험.
  6. AI 도구로 글 만들 때 기준. Search Essentials + spam policy 충족이 전제. AI 사용 자체보다 결과물의 helpful/reliable/people-first 여부가 판단 기준.
  7. 자기 점검 질문. “내 방문자가 이 콘텐츠를 만족스럽다고 느낄까?” Yes 면 진행.

§ 04 방법 — 기술 구조

  1. Search 기술 요건 충족. 인덱싱 가능 + 스니펫 노출 가능해야 AI 기능에 등장 가능. 인덱싱·서빙 자체는 보장이 아님.
  2. 크롤링 최적화. 콘텐츠는 공개 크롤 가능해야 함. 대규모·고빈도 갱신 사이트는 crawl budget guide 점검.
  3. 시맨틱 HTML — 인간 가독성 우선. 완벽한 시맨틱 강박 불필요. 접근성 사용자를 위해 가능한 한 시맨틱 마크업 권장.
  4. JavaScript SEO. JS 안의 콘텐츠도 차단되지 않으면 처리 가능. JavaScript SEO best practices 준수.
  5. 페이지 경험. 모든 디바이스 대응, 지연 감소, 주 콘텐츠와 기타 요소 시각적 구분.
  6. 중복 콘텐츠 축소. 의미 없는 URL 에 크롤 자원이 낭비되지 않도록 정리.
  7. 모니터링. Search Console 인증 후 인덱싱·트래픽·에러 진단.

§ 05 방법 — 로컬 & 이커머스

생성형 AI 응답은 적절한 경우 제품 리스팅, 제품 정보, 로컬 비즈니스 정보를 포함한다.

  • 제품 — Merchant Center / Merchant Center 피드를 통한 정형 데이터 공급.
  • 로컬 — Google Business Profile 정비.
  • 추가 — Business Agent (Search 상의 대화형 브랜드 채팅) 같은 머천트 경험을 사업 성격에 맞게 검토.

§ 06 하지 말아야 할 것 — Mythbusting

항목Google 입장
llms.txt 등 “AI 전용” 마크업 파일 불필요. Google 은 별도 처리하지 않는다.
콘텐츠 강제 chunking 불필요. 한 페이지 안의 여러 주제도 시스템이 nuance 파악. 이상적인 페이지 길이 없음.
AI 시스템 전용 재작성 불필요. 동의어·일반 의미 매칭 가능. long-tail 변형을 모두 잡을 필요 없음.
가짜 mention 만들기 비효과적. 핵심 랭킹은 고품질 콘텐츠 우선, 스팸 시스템이 인위 mention 차단.
구조화 데이터 과집중 AI 검색에 필수 아님. 단 rich result 자격 확보용으로 일반 SEO 차원에서는 계속 유용.

§ 07 Agentic 경험 (Browser Agents)

  • AI agent 는 사람 대신 작업 수행 (예: 예약, 제품 사양 비교).
  • Browser agent 는 사이트 접근 시 시각 렌더링(스크린샷), DOM 구조, 접근성 트리를 함께 해석한다.
  • 대응 방법 — Agent-friendly website best practices 검토.
  • 신규 표준 — Universal Commerce Protocol (UCP) 같은 프로토콜이 Search agent 의 작업 범위를 확장 중.

§ 08 실행 우선순위

  1. 생성형 AI 검색을 별도 분야로 분리하지 말고 기존 SEO 기본기부터 정비.
  2. Non-commodity, people-first 콘텐츠로 차별화.
  3. AEO / GEO hack (llms.txt, chunking, AI 전용 재작성, 가짜 mention) 무시.
  4. Agentic 경험 흐름 (browser agent, UCP) 을 관찰하며 사이트 준비도 점검.

§ 09 DEXA / Hermes 운영 적용 메모

  • DEXA 공개 산출물(블로그·도서·문서)도 같은 원칙: 1차 경험 기반 글(Hermes 운영기, PKM 실패 사례 카탈로그) 이 생성형 검색에서 더 잘 surfacing 됨.
  • Hermes 책 PART I 응용 사례 와 PART J 안정성·거버넌스 는 commodity 화 위험이 큰 영역 — 재작성 시 1차 경험과 구체 수치·실패 모드를 반드시 보존.
  • 공개 원고 안전 규칙(masking, 개인 경로·식별자 제거)은 그대로 유지. Google 가이드의 “people-first” 와 충돌하지 않음.
