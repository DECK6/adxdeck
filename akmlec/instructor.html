AKM 강사용 전체 교안

회사 파일 3–5개를 원본 보존, 정본 지식, 검색, 근거 있는 답, 검증과 Learn Back으로 연결하는 120분 수업 운영서입니다. 기술 이름보다 지식의 생애주기와 책임을 먼저 가르칩니다.

120분 진행표 열기수강생 페이지로 돌아가기
핵심 교수 원칙저장소, 정본, 검색, 답 생성, 거버넌스를 서로 대신할 수 없는 층으로 설명합니다. 설명이 끝날 때마다 학습자의 실제 산출물과 통과 신호를 확인합니다.

02 · RUN OF SHOW

120분 전체 진행표

각 행은 의도, 설명, 학습자 행동, 증거와 통과 신호를 함께 가집니다. 시간이 부족해도 보안, 출처, 검증, Learn Back은 생략하지 않습니다.

시간 / 블록왜 필요한가강의 스크립트·핵심 멘트학습자 행동증거통과 신호
0~10분
문제와 ABC 사례		제품 이름보다 조직의 실제 실패를 먼저 봐야 기술이 제자리를 찾습니다.“지금 찾지 못하는 것은 파일입니까, 결정입니까, 근거입니까?” 파일명을 아는 것과 현재 기준을 아는 것은 다르다고 짚습니다.현재 가장 찾기 어려운 회사 질문 하나를 씁니다.질문 카드 1개파일명 검색만으로 답할 수 없는 질문이 하나 나옵니다.
10~25분
지식관리 층위		데이터·문서·지식과 저장·검색·생성·거버넌스를 분리해야 합니다.ABC의 일정 셀→제안서→설치일 정보→재사용 지식→이번 고객 맥락→SOP→운영 기억→승인 증거를 순서대로 연결합니다.자기 회사 자료 하나를 데이터, 문서, 정보, 지식, 맥락, 절차, 기억, 증거로 분해합니다.8칸 분해표원본과 해석, 일반 규칙과 이번 조건, 절차와 증거가 구분됩니다.
25~45분
AKM 7개 레이어		도구를 보기 전에 목표 아키텍처와 변경 규칙을 고정합니다.“Source는 영수증, Knowledge는 여러 영수증으로 만든 현재 판단, Context는 이번 조건, Procedure는 다음 사람이 따라 할 순서, Evaluation은 실제로 맞았는지 본 기록입니다.”준비 파일을 Source, Knowledge, Context, Memory, Procedure, Action, Evaluation에 임시 배치합니다.레이어 맵모든 자료가 한 레이어에 몰리지 않고 파생 노트의 역할이 나뉩니다.
45~60분
파이프라인·검색 사다리		같은 자료가 인입, 정본, 색인, 답, 검증에서 다른 책임을 가집니다.“ID는 grep, 다른 표현은 semantic, 연결 경로는 Graphify, 최종 사실은 원문입니다.” 검색 점수와 진실을 분리합니다.질문 3개를 exact, semantic, relationship, direct read에 매핑하고 이유를 씁니다.검색 선택표각 질문에 왜 그 검색부터 시작하는지 설명합니다.
60~75분
13개 개념 지도		제품을 만능 해결책으로 오인하는 것을 막습니다.각 카드의 “해결하지 못하는 것”을 먼저 읽습니다. Vault≠Vector DB, Qdrant≠RAG, LangGraph≠knowledge graph를 확인합니다.혼동하기 쉬운 짝 3개를 선택해 반례 문장을 만듭니다.오해 방지 문장 3개저장소, 인덱스, 생성 패턴, 앱 프레임워크, 워크플로 런타임을 구분합니다.
75~105분
미니 AKM 실습		개념을 실제 파일 변환, 검색, 답과 개선으로 연결합니다.보안→인입→metadata→정본→Context/Procedure 분리→검색 3종→근거 답→Verify/Learn Back을 한 단계씩 진행합니다.템플릿으로 파일 묶음을 처리하고 source ID가 있는 답 하나를 만듭니다.source note, canonical note, context, procedure, 검색표, 답, 검증표출처가 있는 답 하나와 실제로 고칠 레이어 하나가 나옵니다.
105~115분
도입 의사결정		파일럿 결과를 조직 운영으로 확장하되 과잉 인프라를 막습니다.Phase A–E와 운영 신호를 설명합니다. “문서 수가 아니라 반복되는 실패와 권한·동시성 요구가 선택 조건입니다.”Markdown, semantic index, graph, Qdrant, RAG, 프레임워크를 지금/나중/아직 아님으로 나눕니다.1페이지 도입안필수 기반과 선택 인프라가 분리되고 각 선택에 관찰 가능한 이유가 있습니다.
115~120분
회고·첫 행동		수업 결과를 30일 파일럿으로 고정합니다.다섯 질문을 빠르게 복습하고 “한 팀, 한 주제, 한 소유자, 한 골드 질문”을 지정하게 합니다.담당자, 안전한 파일 묶음, 성공 질문을 하나씩 지정합니다.30일 파일럿 선언소유자, 범위, 성공 질문과 첫 검토 날짜가 모두 있습니다.
90분 압축

핵심 루프는 보존

  1. 0–10분 ABC 문제 정의
  2. 10–25분 지식관리와 7개 레이어
  3. 25–40분 파이프라인과 검색 사다리
  4. 40–55분 혼동 큰 7개 개념 대면 설명, 나머지는 읽기
  5. 55–80분 3개 파일·1개 질문 실습
  6. 80–90분 도입 판정과 30일 계획

절대 삭제하지 않음: 보안, 출처, 검증, Learn Back.

180분 확장

비교 실험과 운영 훈련

120분 기본안 뒤에 60분을 추가합니다.

  1. 20분: 실제 파일의 Chunking·metadata 비교
  2. 15분: 동일 질문으로 exact·semantic·graph 결과 비교
  3. 15분: ACL과 출력 유출 시나리오 테이블탑
  4. 10분: 골드 질문과 평가표 공동 작성

완료 증거: 비교표, 권한 HOLD 기록, 골드 질문 세트.

03 · NARRATIVE

ABC Studio 교수 내러티브와 공개 순서

문제를 한꺼번에 설명하지 말고, 학습자가 “폴더 정리로 충분하다”는 가설을 스스로 수정하도록 순서대로 공개합니다.

파일 다섯 개만 보여줍니다

회의 전사, 제안서, 프로젝트 표, SOP, 결정 로그. 먼저 “어디에 저장할까?”를 묻습니다.

확인 질문: 각 파일에서 직원이 정말 찾고 싶은 것은 무엇입니까?

Drive와 채팅의 한계를 드러냅니다

최종_v3_진짜최종 중 현재 기준, 회의와 제안서의 불일치, 상태를 바꾼 승인, SOP 미반영 실패를 차례로 묻습니다.

교수 포인트: 폴더는 위치를 개선하지만 의미, 출처, 신뢰도, 소유자, 검토일을 만들지 않습니다.

권한 문제를 마지막에 추가합니다

검색은 잘 되지만 볼 수 없는 자료가 답에 섞인 상황을 제시합니다.

교수 포인트: ACL은 원본 저장소뿐 아니라 index, retrieval, context, output까지 유지되어야 합니다.

원본을 그대로 보존합니다

원본 다섯 개를 고치지 않고 Source로 둡니다. 정규화, 추출, 요약은 별도 파생물입니다.

교수 포인트: provenance가 없으면 정본이 틀렸을 때 되돌아갈 수 없습니다.

다섯 파생물을 만듭니다

고객 요구 정본, 프로젝트 맥락, 설치 SOP, 결정 로그, 검증 기록. 각 문서는 역할과 변경 규칙이 다릅니다.

확인 질문: 이번 고객만의 조건을 일반 Knowledge로 만들면 어떤 문제가 생깁니까?

검색과 근거 있는 답을 연결합니다

ID는 exact, 표현이 다른 요구는 semantic, 결정→SOP는 relationship, 최종 판단은 direct read로 확인합니다.

교수 포인트: 후보 회수와 사실 판정은 분리합니다.

실패를 다음 실행에 반영합니다

잘못된 답의 원인을 source, chunk, metadata, knowledge, procedure, query로 분류하고 실제 책임 레이어를 수정합니다.

마무리 문장: “지식관리체계는 파일에서 합의·절차·근거·책임을 꺼내 다음 실행에 다시 넣습니다.”

04 · LAYERS

AKM 7개 레이어: 설명과 회사 예시

레이어는 폴더명이 아니라 역할, 신뢰도, 변경 규칙입니다. Output과 Archive는 외부 전달과 현행성 경계를 돕는 보조 구역입니다.

01 · Source원본 보존

회의 전사, PDF, 표, 결정 기록.

원본은 수정하지 않고 출처·무결성·권한을 남깁니다.
02 · Knowledge재사용 판단

고객 요구 정본, 설치 원칙, 비교·용어.

출처, 범위, 모순, 신뢰 상태를 함께 관리합니다.
03 · Context이번 조건

프로젝트 일정, 고객 제약, 현재 담당.

특정 상황을 일반 지식과 분리합니다.
04 · Memory짧은 포인터

매번 먼저 볼 안정적인 운영 규칙.

긴 전문과 자주 변하는 상태를 넣지 않습니다.
05 · Procedure반복 실행

제안 검토, 설치, 장애 복구 체크리스트.

단계, 실패, 증거, 판정이 있어야 합니다.
06 · Action실행 기록

결정, 변경, 인계, 재현에 필요한 기록.

모든 호출이 아니라 재현과 책임에 필요한 것만 둡니다.
07 · Evaluation검증과 개선

골드 질문, 감사표, 실패 패턴.

실패를 책임 레이어 수정으로 되돌립니다.
Output

외부 전달물

보고서, 교육자료, 납품물. 공유 전 이름, 링크, 표의 숨은 열, 첨부 metadata를 다시 확인합니다.

Archive

현행이 아닌 자료

대체된 SOP와 종료 프로젝트 정본. 삭제로 잊히게 하지 말고 deprecated 상태와 대체 정본을 가리킵니다.

세 가지 관련 개념의 자리

CMDS

Connect → Merge → Develop → Share. 지식 생애주기의 프로세스 문법.

LLM Wiki

raw sources → compiled wiki → schema/index/log. Knowledge 레이어를 강하게 만드는 컴파일 패턴.

AKM

레이어를 실행·검증·Learn Back으로 연결하는 전체 운영 아키텍처.

변경 규칙

Source는 불변, Knowledge는 새 근거로 갱신, Context는 상태에 따라 갱신, Procedure는 반복 실패와 검증으로 개선, Evaluation은 실패와 수정 대상을 기록합니다. 한 규칙 전문은 한 정본에 두고 다른 곳에는 포인터만 둡니다.

05 · 13 CARDS

13개 개념 심화 카드

각 카드는 정의, 비유, 작동 원리, 해결 범위, 장점과 위험, 언제 필요한지, 최소 사용과 확장, AKM 상태·매핑, 강사 포인트와 회사 예시까지 동일한 틀로 설명합니다.

AKM · Agent Knowledge Management

core
정의
원본, 합성 지식, 맥락, 운영 기억, 절차, 실행 기록, 평가를 분리 저장하고 Learn Back으로 연결하는 운영 아키텍처.
비유
자료실, 편집실, 매뉴얼, 실행 일지, 품질관리실이 연결된 회사.
작동 원리
Schema로 역할을 기술하고 Router로 분류하며 Ingest→Classify→Compile→Contextualize→Execute→Verify→Learn Back을 반복.
해결
원본·해석 혼합, 맥락 유실, 반복 재해석, 절차 누락, 검증 없는 완료, 동일 실패 재발.
해결하지 않음
잘못된 원본, 약한 책임 구조, 조직 정치, 무단 접근, 검토하지 않는 문화를 자동 해결하지 않음.
강점
제품이 바뀌어도 지식의 생애주기와 책임 구조를 유지.
위험·운영 부담
과도한 세분화와 오라우팅. Evaluation이 약하면 복잡한 폴더 체계로 퇴행.
필요 시점
여러 팀이 같은 파일을 반복 해석하거나 같은 실수가 재발하고 AI 답을 재사용하려 할 때.
최소 사용
Source, Knowledge, Context, Procedure, Evaluation와 index, router, 검증표.
확장
도메인 owner, 권한 등급, 검색 인덱스, 감사 로그, 평가 세트, deprecation.
AKM 상태·매핑
Core. 7개 레이어, Schema, Router, Loop, Security, Index/Log, query policy.
강의 포인트
“AKM은 제품이 아니라 책임과 흐름의 설계.”

Chunking · 청킹

supporting
정의
긴 문서를 검색과 컨텍스트 조립에 적합한 의미 단위로 나누는 처리.
비유
책을 제목과 문맥을 잃지 않는 카드 묶음으로 만드는 일.
작동 원리
고정 길이·overlap, 문단/heading 구조, 의미 변화, parent-child, metadata 보존 방식을 문서 유형에 맞게 선택.
해결
긴 전사·정책·보고서에서 관련 구간을 찾고 모델에 필요한 부분만 제공.
해결하지 않음
폴더 분류, 정본 설계, 신뢰도, 권한, 출처 검증.
강점
검색 정밀도와 컨텍스트 비용을 조절.
위험·운영 부담
너무 작으면 조건과 결론 분리, 너무 크면 잡음 증가. overlap·중복 비용, metadata 유실, 재색인 부담.
필요 시점
긴 PDF, 회의 전사, 정책집, 코드 문서를 semantic/RAG 검색에 넣을 때.
최소 사용
heading과 문단을 보존하고 원본, 섹션, 날짜, ACL을 chunk에 부착.
확장
타입별 splitter, parent-child, 표/이미지 파싱, chunk 평가와 재색인.
AKM 상태·매핑
Supporting. qmd와 RAG 인덱싱 단위이며 canonical note·폴더 분류와 별개.
강의 포인트
경계 하나가 검색 품질과 답의 범위를 바꾼다고 강조.
회사 예시
“예산 변경”만 잘리고 앞의 “승인 전제”가 빠지면 잘못된 확정 답이 됨.

optional
정의

최소 사용

확장
전용 DB, hybrid retrieval, reranking, sharding, replication, 관측성.
AKM 상태·매핑

강의 포인트
“유사도는 후보 회수 점수이지 사실 점수가 아닙니다.”
회사 예시

RAG · 검색증강생성

supporting
정의
질문 관련 자료를 검색·선별·조립해 모델이 답하고 출처와 범위를 검증하는 패턴.
비유

필요 시점

최소 사용
골드 질문 10개, top-k, 원문 인용, 답 검증표.
확장
hybrid, reranking, query rewriting, access-aware retrieval, tracing, 평가 세트.
AKM 상태·매핑
Supporting query pattern. 좋은 합성은 검토 후 Knowledge·Procedure·Evaluation에 file-back.
강의 포인트
인용이 있다는 사실과 인용이 주장을 지지한다는 사실을 분리.
회사 예시

06 · Q&A

예상 질문과 강사 답변

짧게 답한 뒤 항상 “우리 조직의 source, owner, 권한, 골드 질문은 무엇인가?”로 되돌립니다.

Q1. Notion이나 Confluence로도 AKM을 만들 수 있습니까?

가능합니다. AKM은 tool-agnostic입니다. 원본·정본·맥락·절차·평가의 분리, metadata, ACL, provenance, export 가능성, direct verification이 중요합니다. Obsidian은 현재 구현의 한 선택입니다.

Q2. Vector DB를 도입하면 검색이 모두 좋아집니까?

아닙니다. embedding model, chunk boundary, metadata, hybrid search, reranking, query type, source freshness가 함께 맞아야 합니다. 이름·ID·날짜는 exact search가 더 낫습니다.

Q3. RAG 답에 citation이 있으면 믿어도 됩니까?

인용은 검증의 출발점입니다. source가 주장을 직접 지지하는지, 최신인지, 권한이 맞는지 확인해야 합니다. 인용이 있어도 범위가 과장될 수 있습니다.

Q4. Graphify가 자동으로 ontology를 만들어 줍니까?

현재 문서 graph는 wikilink와 tag라는 명시 관계를 추출합니다. 관계 타입과 entity identity가 충분하지 않으면 완성된 ontology가 아닙니다. 구조 탐색 신호이지 조직 진실 판정기가 아닙니다.

Q5. GBrain을 바로 회사에 설치하는 것이 빠르지 않습니까?

강한 비교 대상이지만 회사 schema, source ownership, ACL, 골드 질문 없이 도입하면 복잡한 시스템이 먼저 생깁니다. 격리된 비기밀 sandbox에서 중복, 운영비, leak test를 비교한 뒤 결정합니다.

Q6. 좋은 답변을 모두 Knowledge에 저장해야 합니까?

아닙니다. 재사용 가치, 출처, 맥락, 검증이 있는 답만 file-back합니다. 일회성 조회와 임시 초안은 저장하지 않아 오래된 쓰레기 지식이 쌓이지 않게 합니다.

Q7. AI가 정본을 자동 수정하게 해도 됩니까?

AI는 초안과 변경 제안을 만들 수 있지만 Source는 수정하지 않고 중요한 canonical change는 domain owner가 검토합니다. 자동 write는 범위, diff, rollback, audit를 포함해야 합니다.

Q8. 언제 LangGraph가 필요합니까?

체크리스트로도 운영 가능한 초기에는 필요하지 않습니다. 장기 실행, 상태 저장, 승인 대기, 재시도, 분기와 복구를 서비스로 운영할 때 검토합니다.

10 · GOVERNANCE

회사 역할, 보안, 정본과 보존

검색 기술이 좋아도 책임·권한·현행성 규칙이 없으면 회사 지식은 다시 문서 더미가 됩니다.

역할책임금지선
Contributor원본과 현장 지식을 source ID, 보안 상태와 함께 제출.확인되지 않은 해석을 정본처럼 올리지 않음.
Knowledge Stewardschema, provenance, 중복, 링크, 검토 주기, deprecation 운영.모든 도메인 판단을 혼자 승인하지 않음.
Domain Owner내용의 정확성, 적용 범위, 최신성, 정본 변경 승인.기술·권한 운영을 방치하지 않음.
Reviewer출처, 논리, 위험, 공유 가능성, claim support 검증.링크가 있다는 이유만으로 승인하지 않음.
Consumer / Agent허용 범위에서 검색·사용하고 gap과 오류 보고.검색 결과를 원문 확인 없이 재배포하지 않음.

SSOT와 provenance

SSOT는 파일 하나만 존재한다는 뜻이 아니라 현재 승인 판단을 찾을 수 있다는 뜻입니다. 주장마다 원본 계보와 대체 관계를 둡니다.

ACL과 permissions

storage, normalization, index, retrieval, context assembly, 생성 답, output에서 같은 접근 범위를 강제합니다. 다른 역할로 leakage regression을 실행합니다.

Review와 trust

raw, draft, unverified, reviewed, canonical, deprecated를 구분하고 owner, reviewer, review due를 기계가 읽게 합니다.

보안 선검사

분류 전에 비밀 값과 불필요한 개인정보를 제거합니다. 인입 거절은 실패가 아니라 안전한 정상 경로입니다.

Retention

원본, 로그, 임시 추출, 답변, 출력의 보존 기간과 파기 책임을 업무·규정·비용에 맞게 다르게 둡니다.

Deprecation

오래된 지식을 조용히 지우지 않습니다. deprecated 상태, 대체 정본, 변경 이유, 기본 검색 제외를 기록합니다.

출력 유출 표면

본문뿐 아니라 링크 대상, 인용문, 표의 숨은 열, 주석, 첨부파일과 이미지 metadata까지 확인합니다. 출력 검토가 끝나기 전에는 외부 전달을 HOLD로 유지합니다.

11 · ADOPTION

30·60·90일, 실패 복구와 다음 산출물

한 팀의 작은 루프를 검증한 뒤 검색·관계·서비스를 붙입니다. 같은 실패가 반복되면 사람의 주의가 아니라 책임 레이어를 바꿉니다.

0–30일

File-first foundation

  • 한 팀·한 주제·원본 20–50개
  • source inventory·보안·owner
  • 정본 5–10개·review state
  • exact search·direct read
  • 골드 질문 10–20개
  • 원본→답→검증 한 loop
31–60일

Retrieval ladder

  • semantic 후보 검색
  • 의미 있는 링크·Graphify 탐색
  • source-to-claim·freshness
  • SOP 2–3개·failure pattern
  • miss·false positive 기록
  • 권한 역할별 시험
61–90일

Service governance

  • 인증 query/RAG pilot
  • citation·gap reporting
  • access leakage regression
  • latency·cost·freshness·recall
  • 선택 인프라 필요성 판정
  • review·deprecation cadence

13 · CHECKLIST

수업 전·후 강사 체크리스트

수업의 성공은 슬라이드 진도가 아니라 안전한 파일로 작은 지식 루프를 끝까지 돌리고 실제 개선 대상을 남겼는지로 판정합니다.

수업 전

  • 수강생 페이지와 이 페이지를 로컬/온라인에서 열어봤다.
  • ABC Studio 샘플 5종이 실제 조직과 무관한 가상 자료다.
  • 학습자 파일에 privacy gate를 먼저 적용할 시간을 확보했다.
  • Markdown 편집기와 exact 검색이 작동한다.
  • 복사 버튼과 fallback을 확인했다.
  • 질문 3개와 기대 source를 준비했다.
  • 90분 압축 여부를 시작 전에 결정했다.
  • HOLD 자료를 대체할 안전 샘플이 있다.
  • 공개 출력에 내부 정보가 없는지 확인했다.

수업 후

  • 7개 기대 산출물이 존재한다.
  • 근거 있는 답의 핵심 주장에 source ID가 있다.
  • 원본과 정본, Context와 Procedure가 분리되었다.
  • 세 검색 route의 선택 이유가 기록되었다.
  • 권한 밖 정보가 출력에 없음을 확인했다.
  • PASS/HOLD/FAIL 판정과 owner가 있다.
  • Learn Back 대상이 실제로 한 곳 이상 수정되었다.
  • 30일 파일럿의 팀, 주제, owner, 골드 질문이 정해졌다.
  • 검토 날짜와 다음 산출물이 정해졌다.
최종 수업 판정

PASS: 구조·출처·실습·보안·다음 행동이 모두 있음. HOLD: owner 승인이나 근거 보완이 필요해 공유를 멈춤. FAIL: 실제 정보 유출, 원본 훼손, 출처 없는 정본, 요구 산출물 누락이 있음.
