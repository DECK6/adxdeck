120분 · 회사 지식관리 실전 과정

AKM — 회사 지식이 다시 일하게 만드는 법

흩어진 업무 파일을 더 많이 쌓는 대신, 원본과 정본을 구분하고 맥락·절차·검색·검증을 연결해 다음 일에 다시 쓰는 방법을 배웁니다.

오늘 우리가 만들 것 내 회사 파일로 해보기
120분설명과 실습
3–5개안전하게 정리한 파일
1개근거 있는 답과 개선점
도구보다 생애주기를 먼저 설계합니다. 파일이 어디에서 왔고, 무엇이 현재 기준이며, 누가 확인했고, 실패를 어디에 반영할지 정하면 기술은 필요한 자리에 놓입니다.
01 · TODAY

오늘 우리가 만들 것

작은 파일 묶음을 보안 확인부터 출처가 있는 답, 검증과 개선까지 한 바퀴 돌립니다. 제품 설치보다 “왜 이 정보가 이 자리에 있어야 하는가”를 말할 수 있는 것이 목표입니다.

수업이 끝나면 나는…

원본·정본·맥락·절차·증거의 차이를 설명할 수 있습니다.

질문에 맞게 exact·semantic·relationship·direct read를 고를 수 있습니다.

13개 개념의 역할과 한계를 구분할 수 있습니다.

회사 파일 3–5개를 미니 AKM으로 변환할 수 있습니다.

답의 핵심 주장에 출처를 붙이고 검증할 수 있습니다.

30·60·90일 도입 순서를 제안할 수 있습니다.

전체 120분 학습 여정

시간무엇을 배우나내가 하는 일남는 결과다음으로 갈 기준
0–10분ABC Studio의 “파일은 많지만 기억은 없는” 문제지금 가장 찾기 어려운 회사 질문 하나를 적습니다.질문 카드 1개파일명만으로 답할 수 없는 질문이 나옵니다.
10–25분데이터·문서·정보·지식·맥락·절차·기억·증거회사 자료 하나를 여덟 역할로 분해합니다.자료 분해표원본과 해석, 절차와 증거가 구분됩니다.
25–45분AKM 7개 레이어와 Learn Back 루프준비 파일을 레이어에 임시 배치합니다.레이어 지도모든 것이 한 폴더나 한 타입에 몰리지 않습니다.
45–60분전체 파이프라인과 검색 사다리질문 세 개에 검색 방식을 붙입니다.검색 선택표왜 이 검색을 골랐는지 설명할 수 있습니다.
60–75분13개 핵심 개념과 혼동 방지헷갈리는 짝 세 개로 반례를 만듭니다.오해 방지 문장저장소·검색·생성·워크플로를 구분합니다.
75–105분3–5개 파일 미니 AKM 실습인입→분류→정본→검색→답→검증을 수행합니다.정본·절차·근거 있는 답출처가 있는 답과 고칠 레이어가 하나씩 나옵니다.
105–115분지금/나중/아직 아님 도입 판정우리 조직의 운영 신호를 표시합니다.1페이지 도입안필수 기반과 선택 인프라가 분리됩니다.
115–120분회고와 첫 30일 행동담당자·파일 묶음·검증 질문을 정합니다.30일 파일럿 선언소유자, 범위, 성공 질문이 모두 있습니다.
02 · CASE

ABC Studio: 파일은 많지만 기억은 없다

가상의 전시 제작사 ABC Studio는 Drive와 채팅에 업무가 흩어져 있습니다. 실습에서는 실제 고객이나 사람을 쓰지 않고 이 안전한 사례로 구조를 먼저 익힙니다.

회의 전사

요구, 보류, 발언은 있지만 “최종 승인 범위”가 따로 없습니다.

제안서

목표, 일정, 예산, 콘셉트는 있지만 이후 변경과 연결되지 않습니다.

프로젝트 표

담당자와 상태는 있지만 상태를 바꾼 결정 근거가 보이지 않습니다.

SOP

설치와 장애 대응 순서는 있지만 이번 실패가 반영됐는지 모릅니다.

결정 로그

채팅과 이메일 속 변경·승인·취소가 현재 기준과 분리되어 있습니다.

Drive와 채팅만 있을 때 → 출처가 있는 회사 지식

Before · 위치만 찾기

  • 무엇이 진짜 최신인지 모릅니다.
  • 합의와 근거가 여러 형식에 흩어집니다.
  • 실패가 다음 SOP로 이어지지 않습니다.
  • 검색 결과의 권한 경계가 불명확합니다.
→

After · 다음 실행에 쓰기

  • 원본은 보존하고 정본은 따로 합성합니다.
  • 결정마다 source ID와 적용 범위를 둡니다.
  • 실패를 절차·평가 규칙에 반영합니다.
  • 검색부터 답까지 같은 권한을 유지합니다.
핵심 질문

“파일이 어디 있지?”에서 멈추지 말고 “현재 승인된 판단은 무엇이며, 어떤 원본이 지지하고, 다음 사람이 무엇을 해야 하지?”까지 물어봅니다.

03 · LANGUAGE

같아 보이지만 역할이 다른 여덟 가지

한 문서에 모두 섞어두면 최신성, 적용 범위, 실행 조건과 증거를 잃습니다. 다음 질문에 답해보세요: “내가 가진 것은 원본인가, 현재 판단인가, 이번 프로젝트만의 조건인가?”

01

데이터

해석 전의 값이나 기록. 일정 날짜, 비용 셀, 상태값.

02

문서

데이터를 형식으로 묶은 파일. PDF, 표, 발표자료, 메시지 기록.

03

정보

질문에 맞게 해석한 데이터. “설치일은 8월 3일.”

04

지식

여러 정보에서 재사용할 판단을 합성한 것. 출처와 범위가 필요합니다.

05

맥락

특정 조직·고객·프로젝트에서만 참인 조건과 현재 상태.

06

절차

반복 실행할 순서, 실패 처리, 완료 증거와 판정 기준.

07

기억

매번 먼저 알아야 하는 짧고 안정적인 규칙 또는 포인터.

08

증거

주장·결정·완료를 확인하는 원본 위치, 승인 기록, 검증 결과.

저장소에서 운영체계까지

층하는 일혼자 해결하지 못하는 일
파일 저장소원본, 버전, 기본 권한을 보관합니다.흩어진 판단을 현재 기준으로 합성하지 않습니다.
위키사람과 AI가 읽는 정본 지식을 연결합니다.원본 보존, 검색 품질, 권한 전체를 대신하지 않습니다.
검색 인덱스exact·semantic·relationship 후보를 찾습니다.후보가 참인지, 최신인지 결정하지 않습니다.
답변 생성근거 후보를 질문에 맞는 답으로 조립합니다.인용 적합성이나 과장을 자동 보장하지 않습니다.
거버넌스소유, 권한, 검토, 보존, 폐기, 감사를 정합니다.기술이 사람의 책임을 대신하지 않습니다.
04 · MAP

AKM 7개 레이어와 두 개의 보조 구역

AKM은 특정 앱 이름이 아닙니다. 서로 다른 신뢰도와 변경 규칙을 가진 지식을 분리하고, 실행과 검증으로 다시 연결하는 도구 독립적 운영 아키텍처입니다.

06 · CONCEPTS

13개 핵심 개념 카드

한 줄 정의만 외우지 말고 “무엇을 해결하고, 무엇은 해결하지 않는가”를 함께 봅니다. 회사에서 쓸 때의 최소 범위와 AKM 안의 자리를 확인하세요.

AKM

core
한 줄 정의
에이전트가 원본, 지식, 맥락, 운영 기억, 절차, 실행, 평가를 분리하고 Learn Back으로 연결하는 운영 아키텍처.
비유
자료실·편집실·매뉴얼·실행 일지·품질관리실이 연결된 회사 운영체계.
해결
원본과 해석의 혼합, 맥락 유실, 반복 재해석, 검증 없는 완료, 같은 실패의 재발.
해결하지 않음
잘못된 원본, 책임 회피, 무단 접근, 검토하지 않는 문화를 자동으로 고치지 않습니다.
회사 사용
Source·Knowledge·Context·Procedure·Evaluation 최소 다섯 영역과 분류 규칙, 검증표로 시작합니다.
AKM 상태
Core. 다른 모든 도구의 저장 위치, 책임, 검증과 개선 방식을 정합니다.

Chunking

supporting
한 줄 정의
긴 문서를 검색과 컨텍스트 조립에 맞는 의미 단위로 나누는 처리.
비유
책을 통째로 넣지 않고 제목과 문맥을 잃지 않는 카드 묶음으로 만드는 일.
해결
긴 회의·정책·보고서에서 관련 구간을 찾고 필요한 부분만 모델에 넣습니다.
해결하지 않음
폴더 분류, 정본 설계, 권한, 신뢰도, 출처 검증을 대신하지 않습니다.
회사 사용
heading과 문단을 보존하고 원본 ID, 섹션, 날짜, 권한 등급을 각 조각에 붙입니다.
AKM 상태
Supporting. qmd와 RAG의 처리 단위지만 저장 레이어와는 별개입니다.

Vector DB

optional
한 줄 정의
임베딩 벡터와 metadata를 저장하고 유사도와 필터로 가까운 항목을 찾는 데이터베이스.
비유
단어가 달라도 뜻이 비슷한 카드를 가까운 서랍에서 찾는 창고.
해결
표현이 다른 의미 검색, 추천, 대규모·멀티모달 유사도 검색.
해결하지 않음
위키, SSOT, RAG 생성, 출처 진실성, 절차와 거버넌스가 아닙니다.
회사 사용
먼저 작은 semantic index로 골드 질문을 시험하고 source ID와 ACL metadata를 보존합니다.
AKM 상태
Optional. 의미 검색 규모와 동시성 요구가 실제 지표로 나타날 때 검토합니다.

RAG

supporting
한 줄 정의
관련 자료를 검색·선별해 컨텍스트로 만들고, 답을 생성한 뒤 인용과 범위를 검증하는 패턴.
비유
사내 자료실에서 서류를 꺼내 책상 위에 놓고 보고서를 쓰는 직원.
해결
최신 또는 사내 자료를 근거로 자연어 답과 보고서를 반복 생성합니다.
해결하지 않음
검색 누락, 오래된 자료, 권한 오류, 모순, 과장, 정본 축적을 자동 해결하지 않습니다.
회사 사용
골드 질문, top-k 후보, source ID 인용, 답 검증표를 갖춘 작은 파일럿으로 시작합니다.
AKM 상태
Supporting. AKM의 검색·답변 구간이며 전체 생애주기는 아닙니다.

Graphify

supporting
한 줄 정의
코드 구조나 문서의 명시 링크·태그를 그래프로 만들어 경로, 허브, 커뮤니티를 보는 로컬 도구.
비유
내용의 유사도보다 어떤 문서가 무엇을 직접 가리키는지 보여주는 노선도.
해결
결정→요구→SOP 경로, 허브, 연결된 이웃, 고립 문서를 찾습니다.
해결하지 않음
semantic 검색, 출처 진실성, 완성된 ontology, 답변 생성을 해결하지 않습니다.
회사 사용
정본에 의미 있는 링크를 만들고 허브와 고립 노드부터 점검합니다.
AKM 상태
Supporting. 명시 관계 탐색이며 Graphiti나 Vector DB가 아닙니다.

Obsidian

core interface
한 줄 정의
로컬 Markdown을 편집하고 링크, backlinks, properties로 탐색하는 지식 작업 인터페이스.
비유
창고 자체가 아니라 Markdown 창고를 사람이 읽고 고치는 작업대.
해결
사람이 읽을 수 있는 노트 편집, 링크 탐색, plain-text 소유와 도구 연동.
해결하지 않음
AKM 레이어, 라우팅, 검증, 권한, RAG를 자동 설계하지 않습니다.
회사 사용
기본 Markdown, properties, internal links, templates, search만으로 시작합니다.
AKM 상태
Core interface. 정본을 편집하는 선택 UI이며 AKM 자체는 아닙니다.

Obsidian Vault

core boundary
한 줄 정의
Obsidian이 노트, 설정, 자산을 읽는 로컬 파일 시스템 폴더.
비유
회사 전체가 아니라 문서가 들어 있는 특정 서고 폴더.
해결
파일의 물리 경계, 백업·동기화·버전관리 단위를 제공합니다.
해결하지 않음
Vector DB, semantic index, RAG, 승인 체계가 아닙니다.
회사 사용
한 팀 파일럿에서 root policy와 source/knowledge/context 경계를 명확히 둡니다.
AKM 상태
Core canonical storage boundary. 검색 인덱스는 Vault를 읽어 별도로 만듭니다.

GBrain

sandbox-only
한 줄 정의
Markdown 정본과 DB, hybrid retrieval, 합성 답, typed graph, MCP와 평가를 묶은 database-backed brain 시스템.
비유
서고 옆에 검색 사서, 분석가, 관계 지도, 권한 게이트를 함께 둔 별도 서비스.
해결
여러 source와 사용자의 scoped 검색·합성·graph·MCP 제공.
해결하지 않음
회사의 분류 기준, 검토 문화, 내용 책임을 자동 확정하지 않습니다.
회사 사용
비기밀 sandbox에서 citation, gap reporting, 권한 격리를 검증한 뒤 비교합니다.
AKM 상태
Sandbox-only reference. AKM의 도구 독립 레이어 모델을 대체하지 않습니다.

Qdrant

not adopted
한 줄 정의
vector와 payload metadata를 point로 저장하고 similarity, filter, hybrid query를 수행하는 Vector DB 제품.
비유
의미 좌표와 업무 표찰을 함께 붙인 대규모 카드 창고.
해결
semantic/vector 검색, metadata filter, dense+sparse hybrid, 서비스형 인덱스 운영.
해결하지 않음
정본 위키, RAG 답, 지식 거버넌스와 출처 검증 자체가 아닙니다.
회사 사용
pilot collection에 source ID, chunk ID, permission group, trust level을 payload로 둡니다.
AKM 상태
Not adopted. 로컬 검색 한계가 반복 지표로 증명될 때만 검토합니다.

grep / ripgrep

supporting
한 줄 정의
파일에서 exact string이나 정규식과 일치하는 행과 위치를 결정적으로 찾는 텍스트 검색 도구.
비유
“이 번호와 문구가 정확히 적힌 카드를 모두 가져오라”는 색인 검색.
해결
ID, 날짜, 파일명, property, 상태, 정확 승인 문구를 빠르게 찾습니다.
해결하지 않음
표현이 다른 의미 유사성, 관계 경로, 답 생성, 진실성 판정을 해결하지 않습니다.
회사 사용
pattern과 검색 범위를 남기고 ignore·hidden 조건으로 누락되지 않았는지 확인합니다.
AKM 상태
Supporting core retrieval. known ID와 exact claim의 첫 선택입니다.

LangGraph

not adopted
한 줄 정의
장시간 실행되는 stateful agent workflow를 지속 실행, 상태 저장, 승인과 재시도로 구성하는 런타임.
비유
지식 지도가 아니라 승인·분기·재시도를 기억하는 업무 프로세스 엔진.
해결
multi-step flow, 재시작, 승인 대기, 상태 분기, 장기 실행.
해결하지 않음
knowledge graph, Vector DB, 정본 지식 저장소, 문서 품질을 제공하지 않습니다.
회사 사용
체크리스트로 먼저 검증하고 반복·실패 상태가 명확한 흐름 하나만 자동화합니다.
AKM 상태
Not adopted. AKM Action 레이어의 선택 런타임입니다.

LlamaIndex

not adopted

비유
자료를 검색 가능한 형태로 가공해 AI에게 건네는 데이터 연결·검색 공구함.
해결
다양한 데이터 형식의 ingestion, parsing, indexing, query/RAG 앱 개발 속도.
해결하지 않음

회사 사용

AKM 상태
Not adopted. Source→retrieval→RAG 구현의 선택 프레임워크입니다.

LangChain

비유
모델과 회사 도구를 연결해 경계가 있는 디지털 작업자를 조립하는 공구함.
해결
여러 model provider, tool, routing, guardrail을 공통 인터페이스로 묶습니다.
해결하지 않음
database, vector store, 정본 노트와 지식 거버넌스 자체가 아닙니다.
회사 사용

Not adopted. Context·Procedure·Retrieval을 앱으로 묶는 선택지입니다.
07 · CLARITY

헷갈리지 않기

이름이 비슷해도 “저장”, “검색”, “생성”, “관계”, “업무 상태”는 다른 층입니다.

혼동정확한 구분기억 문장
Obsidian ≠ AKMObsidian은 Markdown 작업 인터페이스, AKM은 운영 아키텍처입니다.앱을 설치해도 책임 구조는 생기지 않습니다.
Vault ≠ Vector DBVault는 로컬 폴더, Vector DB는 embedding 검색 인프라입니다.서고와 의미 검색 엔진은 다릅니다.
Qdrant ≠ RAGQdrant는 Vector DB, RAG는 검색 결과로 답을 만드는 패턴입니다.창고가 보고서를 쓰지는 않습니다.
Graphify ≠ Graphiti여기의 Graphify는 로컬 관계 그래프 도구이며 Zep Graphiti가 아닙니다.이름이 비슷해도 제품과 구조가 다릅니다.
LangGraph ≠ knowledge graphLangGraph의 graph는 agent workflow와 state transition입니다.업무 흐름과 지식 관계를 구분합니다.
LangChain/LlamaIndex ≠ database둘은 앱 구성·ingestion·retrieval 프레임워크입니다.공구함과 저장소는 다릅니다.
Chunking ≠ folder classificationChunking은 검색 조각 크기, 폴더 분류는 저장 의미입니다.카드 크기와 서가 위치는 다릅니다.
GBrain ≠ AKMGBrain은 특정 brain 구현, AKM은 도구 독립 운영 모델입니다.제품과 운영 아키텍처를 구분합니다.
Graphify ≠ Vector DBGraphify는 명시 관계, Vector DB는 의미 유사도를 찾습니다.연결됨과 비슷함은 다릅니다.
RAG ≠ 전체 지식관리RAG는 검색과 답 생성 구간입니다.정본·절차·검증·폐기는 별도입니다.
08 · WORKBOOK

내 회사 파일로 해보기

기밀을 제거한 회의 기록, 제안서, 프로젝트 표, SOP, 결정 로그 중 3–5개를 복사본으로 준비합니다. 원본을 직접 수정하지 않습니다.

시작 전 프라이버시 게이트

개인정보, 실제 고객명, 계약 수치, 비밀 값, 계정 정보가 없습니다.

외부 AI 서비스에 넣어도 되는지 회사 규정을 확인했습니다.

원본이 아닌 실습 복사본을 사용합니다.

공유 결과에 내부 경로와 숨은 metadata가 남지 않게 확인합니다.

Security check

공개·내부·기밀·제한 중 하나를 표시하고, 누가 읽을 수 있는지와 외부 처리 가능 여부를 적습니다.

남길 것: 자료별 보안·사용 가능 표.

Source intake

각 원본에 source ID, 원본 날짜, 제공 역할, 형식, 무수정 원칙을 기록합니다.

남길 것: Source intake 3–5개.

Metadata와 분류

원본은 Source에 두고, 합성할 지식·맥락·절차·평가 후보를 분리합니다.

남길 것: 레이어와 다음 행동 표.

Canonical note

요약에서 멈추지 말고 현재 승인된 주장, source별 근거, 모순, 적용 범위, 검토일, 대체된 판단을 합성합니다.

남길 것: 정본 지식 노트 1개.

Context / Procedure 분리

이번 프로젝트 조건은 Context로, 누구나 반복할 단계·실패·증거는 Procedure로 옮깁니다.

남길 것: 맥락 노트와 체크리스트 각 1개.

세 가지 retrieval test

정확한 ID 찾기, 표현이 다른 요구 찾기, 결정→요구→SOP 관계 따라가기를 각각 시험합니다.

남길 것: 질문·검색 경로·예상 source·누락 기록.

Citation-backed answer

질문에 답하고 핵심 문장마다 source ID와 근거 위치를 붙입니다. 근거가 없으면 모른다고 쓰고 필요한 자료를 지정합니다.

남길 것: 근거 있는 답 1개.

Verify / Learn Back

질문 적합성, 최신성, 권한, 주장 범위, 인용 적합성을 확인하고 source·chunk·metadata·knowledge·procedure·query 중 고칠 곳을 정합니다.

남길 것: 판정과 실제 수정 항목 1개.

7. Mini governance policy

# 미니 지식 거버넌스 정책

1. 원본은 보존하고 제자리에서 수정하지 않는다.
2. 모든 정본에는 owner, reviewer, review date, source IDs를 둔다.
3. 공개·내부·기밀·제한 자료는 분리 색인하거나 강제 ACL로 필터한다.
4. AI 생성 텍스트는 draft 또는 unverified로 시작한다.
5. 링크 목록은 검증이 아니다. source가 주장을 지지해야 한다.
6. 사용 중단 지식은 대체 정본을 가리키고 기본 답에서 제외한다.
7. 반복 실패는 procedure, schema, router, evaluation 규칙을 바꾼다.
8. 공유 출력은 비밀 값, 개인정보, 내부 경로, 숨은 metadata를 확인한다.
9. 확장 전에 골드 질문으로 검색 품질을 시험한다.
10. 도메인 소유자가 최종 승인과 책임을 유지한다.
